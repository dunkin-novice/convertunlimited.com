Short answer

AVIF is the most efficient image format you can practically ship to the web in 2026 — typically 30% smaller than WebP and 50% smaller than JPEG at visually identical quality. Use it for hero images, art-directed photography, and any page where performance matters. Keep JPEG or WebP fallbacks for the small remainder of browsers that cannot decode AVIF, and for non-web pipelines.

By dunkin-novice · Last updated 3 June 2026

What AVIF actually is

AVIF (AV1 Image File Format) stores a single keyframe of AV1 video inside a HEIF-style container. AV1 is the open, royalty-free video codec developed by the Alliance for Open Media — the same codec your browser uses to decode YouTube and Netflix in their most efficient mode. Wrapping an AV1 keyframe as a still image gives you a format with a modern, ML-tuned compression backbone and no licensing fees.

Like HEIC and WebP, AVIF supports lossy and lossless encoding, alpha transparency, wide colour gamuts (Rec. 2020), 10- and 12-bit colour depth, and HDR. Unlike HEIC, it is fully unencumbered by patent royalties, which is why every browser vendor agreed to ship it.

Size and quality, against JPEG and WebP

The often-quoted figures: AVIF lossy is around 30% smaller than WebP and 50% smaller than JPEG at equivalent visual quality. In practice the gap is larger on photographic content with smooth gradients (skies, skin, water) and narrower on flat graphic content with sharp edges (logos, screenshots) — where lossless PNG or lossless WebP often win on file size.

Where AVIF really separates from older formats is at low quality. JPEG below quality 0.5 shows visible 8×8 block boundaries; WebP shows softening; AVIF degrades much more gracefully. If you are aggressively budgeting bytes for hero images on mobile, AVIF gives you a usable image at file sizes where JPEG looks broken.

Browser support in 2026

AVIF decoding is now supported in every current Chromium-based browser (since Chrome 85 in 2020), Firefox (since 93 in 2021), Safari (since 16.4 in 2023), Samsung Internet, and Opera. caniuse.com puts global decoding support at around 95% of users. The remaining ~5% is a long tail of older Android stock browsers, in-app webviews on older OS versions, and a small slice of Safari users on iOS 15 or earlier.

If you serve images via <picture> with an AVIF source and a WebP or JPEG fallback, you cover everyone:

<picture>
  <source srcset="hero.avif" type="image/avif">
  <source srcset="hero.webp" type="image/webp">
  <img src="hero.jpg" alt="...">
</picture>

Modern CDNs and image-CMS services can generate this triplet automatically. If you are doing it manually, this converter produces the AVIF; pair it with the image converter for the WebP and JPEG fallbacks.

The encoding-speed catch

AVIF's main practical limitation is encoding cost. The reference encoder, libaom, can take seconds to tens of seconds per image at high effort settings. Decoding is fast — that's what matters for end users — but encoding is slow enough that you should batch it and cache the output, not encode on every request.

The browser-based encoder in this tool wraps Chromium's image/avif Canvas export when available. It uses a faster effort setting than libaom at maximum, so output files will typically be slightly larger than what a desktop encoder produces. For most web use cases the difference is small. If you need to squeeze the last percent, run a batch through cavif, avifenc, or a serverside image pipeline.

Where AVIF is not the right answer

  • Outside the browser. Many image editors still treat AVIF as a second-class citizen. Photoshop has a plugin path, GIMP supports it via libheif, but the experience is rougher than JPEG or PNG. If a file is destined for a non-web tool, pick a more portable format.
  • Sharp-edged graphics where every byte counts. Logos, line art, screenshots: lossless WebP or PNG often beats AVIF on file size, and PNG remains the universally accepted format for these.
  • Archival. JPEG and PNG are the safest long-term bets. AVIF will probably still be readable in 20 years (AV1 is too widespread to disappear), but JPEG is closer to a true permanent format.
  • Email and document workflows. Outlook and many corporate document tools still expect JPEG or PNG.

How this converter works

The dropzone above accepts JPG, PNG, and WebP input. The browser decodes the input via the standard image pipeline, paints it onto a Canvas, and re-encodes as AVIF via canvas.toBlob('image/avif', ...). The selected file does not leave your device; there is no server-side upload endpoint for this flow. If your browser can decode AVIF but cannot encode it through Canvas (a few earlier Safari builds were in that state), the page shows an unsupported-browser notice and recommends a desktop tool instead.

You can verify the local-only behaviour in DevTools → Network: open the panel before dropping a file and watch for outbound requests during processing.

Frequently asked questions

Is AVIF really smaller than WebP?

On photographs, yes — typically 20–35% smaller at the same visual quality. On flat graphics and small images, the gap narrows or disappears, and lossless WebP or PNG can win. Always compare on your actual content before replacing production assets.

Will AVIF break older browsers?

Not if you serve it via a <picture> element with a WebP or JPEG fallback. The browser picks the first source it can decode and ignores the rest. The remaining ~5% of users without AVIF support will silently get the fallback.

Why is encoding so slow compared to JPEG?

AV1's compression efficiency comes from a much more sophisticated encoding pipeline than JPEG's DCT-and-quantise. That pipeline is computationally expensive at encode time. Decoding, which is what end users experience, is fast.

Does AVIF support animations?

Yes — AVIF can carry an AV1 image sequence as an animation, similar in spirit to animated WebP. For most use cases a real video file (MP4/AV1) is still the better choice, but if you need a single-file animated still, animated AVIF is a viable option.

Can I convert AVIF back to JPEG or PNG?

Yes — the homepage image converter accepts AVIF input and exports JPG, PNG, or WebP. Use that path when you need to re-export an AVIF for a non-web pipeline.
